# CROWN-MCPP 全地图直接运行与规模实验

## 1. 当前结论

仓库内 10、15、20、50、两张 200 和 400 米地图均已绑定独立的 `crown_fleet_profile.json`。直接运行时，CLI 从地图的 `notes.default_fleet_profile` 自动读取舰队，而不再猜测艇体尺寸。

“直接运行”在本文中只表示：无需再手工传入艇数、初始位置、物理艇体或转弯半径，即可进入完整 CROWN-MCPP 流水线。验收仍要求输出覆盖证书、连续冲突校验和有限上界；它不等价于全局最优，也不等价于已经完成论文级统计实验。

|---|---:|---:|---:|---:|---|---:|---:|
| 10×10 | 1 | 4×2 | 1×0.5 | 0.5 | 否 | 30 | 自动 |
| 15×15 | 2 | 4×2 | 1×0.5 | 0.5 | 否 | 60 | 自动 |
| 20×20 | 2 | 4×2 | 1×0.5 | 0.5 | 否 | 60 | 自动 |
| 50×50 | 3 | 8×4 | 1×0.5 | 0.5 | 否 | 90 | 自动 |
| 200×200 mixed | 4 | 20×10 | 1×0.5 | 0.5 | 否 | 60 | 2 |

这些是算法基准配置，不应未经实艇参数标定就当作工程艇型。尤其是 20×10 m 和 32×16 m 的传感器覆盖足迹与 1×0.5 m 艇体组合，只是明确、可复现的仿真假设。

## 3. 运行方式

单图无需再指定舰队参数：

```bash
MPLCONFIGDIR=/tmp/matplotlib-crown PYTHONPATH=src \
python examples/run_crown_mcpp_experiment.py \
  --map maps/static_obstacle_map_400x400_mixed_obstacles/static_obstacle_map_400x400_mixed_obstacles.json \
  --no-render
```

运行所有内置地图：

```bash
MPLCONFIGDIR=/tmp/matplotlib-crown PYTHONPATH=src \
python examples/run_crown_map_suite.py \
  --timeout-seconds 1500 \
  --outputs-root outputs/crown_map_suite \
  --report outputs/crown_map_suite_report.json
```

论文实验使用预算×重复矩阵；驱动会汇总成功率、首解时间、峰值 RSS、makespan 和 Gap：

```bash
MPLCONFIGDIR=/tmp/matplotlib-crown PYTHONPATH=src \
python examples/run_crown_map_suite.py \
  --maps static_obstacle_map_200x200_mixed_obstacles \
         static_obstacle_map_200x200_mixed_obstacles_s_polygon \
         static_obstacle_map_400x400_mixed_obstacles \
  --lns-budgets 60 180 300 \
  --repeats 3 \
  --timeout-seconds 1500 \
  --outputs-root outputs/crown_budget_stability \
  --report outputs/crown_budget_stability_report.json
```

显式手工舰队仍受支持，但必须同时给出 `--vehicle-length` 和 `--vehicle-width`。覆盖足迹是传感器模型，CLI 和 profile loader 均禁止把它回退为物理艇体。

| 地图 | LNS 预算（s） | 结果 | 总耗时（s） | 端到端首解（s） | 峰值 RSS（MB） | 活跃艇 | makespan |

因为首解各阶段按总预算比例分配，更长预算可能在失败的前置构造器上停留更久，所以当前首解时间和最终候选对预算并不单调。60 秒在三张大图上都成功，且明显减少墙钟时间和连接缓存内存，因此被选为工程默认。400 米图的 300 秒结果 makespan 优约 2.23%，说明论文仍须报告预算消融，不能宣称 60 秒在解质量上普遍最优。

## 5. 这些实验能证明什么

已证明的范围：

- 地图与舰队配置可以一一绑定，七张内置地图不再依赖隐含的艇体假设；
- 覆盖足迹、物理艇体和最小转弯半径在配置、输出目录名和报告中是三个不同概念；
- 当前成功产物覆盖所有责任单元，并通过最终连续冲突验证；
- 200/400 规模运行记录准备时间、求解时间、首解时间、峰值内存、上下界和 Gap；
- 失败预算也会写出失败报告，不会从成功率统计中消失。

尚未证明的范围：

- 单次固定随机种子成功不能称为“首解稳定”。论文应至少报告每个地图—预算组合的多次成功率和首解时间分布；
- 当前大图 Gap 超过 90%，且存在多艘空闲艇，不能据此声称 CROWN-LNS 得到较优解；
- 当前内存峰值不随地图边长单调增长，例如 50 米图的峰值高于两张 200 米图。主要规模变量是责任单元、候选模式和被实际展开的连接缓存，而不是地图边长本身；
- `certified_lns_service_bound` 只表示可行上界配有有效服务工作量下界，不表示根列生成已经收敛，更不表示有限模型全局最优。

因此，工程验收口径可以写为“所有内置地图可直接运行并得到连续验证的可行解”；顶刊论文口径仍必须写为“规模可行性已经打通，稳定性统计和解质量提升仍待完成”。

## 6. 本轮关键改动

1. 每张地图新增独立舰队 profile，明确艇数、初始位姿、传感器覆盖足迹、物理艇体、转弯半径、速度/推力限制、返航策略和默认求解预算。
2. 地图通过 `notes.default_fleet_profile` 声明默认舰队，CLI 自动解析并校验 `map_id` 绑定。
3. 手工 CLI 缺少艇体长宽时直接报错；profile loader 同样拒绝缺失物理艇体的配置。
4. 输出目录同时编码 `sensor`、`hull` 和 `rmin`，消除旧 `footprint` 命名歧义。
5. 200/400 米图使用独立的障碍连接栅格分辨率，避免默认覆盖分辨率错误地主导连接搜索。
6. 模式库增加长责任单元的端点多样性，并允许可由传感器从相邻可航水域覆盖的极小责任狭片生成连续认证的观测模式。
7. CROWN-LNS 增加 BCD 连续分块、顺序连通、空间聚类和贪心连通首解构造；8 单元的 15 米图也优先使用确定性 BCD 连续分块。
8. 回归驱动支持所有地图、预算扫描、重复运行、失败报告和聚合统计。
